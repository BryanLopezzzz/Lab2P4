Problema

Las conexiones HTTPS son seguras, pero son encriptadas y provienen problemas como el MITM (Man In The Middle). Sin embargo la petición inicial del browser al servidor, responde en el protocolo de "handshake" para establecer las llaves de encriptación, es vulnerable (pues claramente aún no está encriptado).

Este artículo propone el uso de un "SHS-HTTPS Enforcer" que es un proceso tipo "daemon" que redirije las peticiones antes de que salgan del cliente, para verificar el URL en una lista de múltiples fuentes. Así, sin importar cual browser se esté usando, la petición será verificada antes de dirigirla al destino final.

Otras soluciones

Algunos browser (user agents), como Google Chrome y Firefox han atacado el problema incluyendo listas estáticas de URLs que solo pueden ser accedidas sobre conexiones seguras HTTPS. El problema es que cada browser tiene su lista. Así un URL puede estar en una lista pero no en la otra, y dependiendo del browser que se use la petición inicial podría ser vulnerable.

Experimento

La solución propuesta fue solo implementada usando como "daemon" servidor proxy Squid. Se le experimentó con peticiones hacia URLs que están en la lista de un browser pero no en la de otro. La solución ha sido exitosa, logrando solo conexiones a sitios seguros y previniendo el problema MITM.
